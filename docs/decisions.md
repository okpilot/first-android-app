## Standing decisions (summary)
- **Stack:** Flutter (Dart) client → Android / Web / Linux desktop; trimmed
  self-hosted Supabase (Postgres + PostgREST + GoTrue) on homebase behind Caddy.
- **Method:** emergent — thin vertical slices, YAGNI, schema grows by migration.
- **Conventions:** modeled on the (verified) LMS Plus conventions, scaled down.

## Decision 1: Build with Flutter, one codebase for all platforms (2026-07-07)
**Context:** Wanted an Android app now and a web interface later with minimal duplicated work; user is on Linux.
**Decided:** Use Flutter/Dart — one codebase compiles to Android + Web + Linux desktop (iOS later, needs a Mac). Start with Web + Linux desktop (both run on the dev machine today); defer the Android SDK.
**Principle:** One codebase, many targets — don't build the UI twice.

## Decision 2: Dart is the only app language; web is not TypeScript (2026-07-07)
**Context:** User assumed a web app requires TypeScript.
**Decided:** Write everything in Dart. Flutter compiles Dart → JavaScript for the browser automatically. No TS / JS / Kotlin by hand.
**Principle:** The browser runs JS, but we don't hand-write it — Dart compiles to it.

## Decision 3: Practice project is a disposable light CRM (2026-07-07)
**Context:** Need a concrete thing to build to learn on. User already self-hosts EspoCRM but does not want to reuse it.
**Decided:** Build a light CRM from scratch as a learning vehicle. The product is disposable; learning is the point.
**Principle:** The vehicle serves the learning, not the other way round.

## Decision 4: Build the emergent way (2026-07-07)
**Context:** User builds through discovery; cannot/does not want to specify large features up front — layout and schema should emerge.
**Decided:** Work in thin vertical slices (walking skeleton first). Grow the schema by forward-only migration, one field/table at a time. Apply YAGNI. Every "add X" = smallest working version → review → next slice.
**Principle:** Discover the design by building it, not before.

## Decision 5: Backend = Postgres, served the Supabase way, trimmed + self-hosted (2026-07-07)
**Context:** User's verified conventions are deeply Postgres/RPC/RLS-shaped; a Flutter client can't safely hit raw Postgres. Full self-hosted Supabase is too heavy; an earlier PocketBase idea wouldn't reuse the Postgres muscle memory.
**Decided:** Self-host a **trimmed** Supabase on homebase — Postgres + PostgREST (REST/RPC) + GoTrue (auth), routed through the existing Caddy (no Kong). Skip Realtime/Storage/Studio/imgproxy. ~80–130 MB idle (less if we reuse an existing Postgres). Flutter uses `supabase_flutter`.
**Principle:** Take Postgres's power and Supabase's conventions; leave the weight behind.

## Decision 6: Model conventions on LMS Plus — verified, then scaled down (2026-07-07)
**Context:** LMS Plus is a mature project with established conventions; a single extraction pass idealized several DB claims.
**Decided:** Adopt LMS Plus conventions, but only after two independent audit rounds verified them. Corrections: it is NOT "everything is RPC" (RPC for multi-table/immutable/sensitive; direct RLS access otherwise); pagination is LIMIT/OFFSET (default 10), not keyset; hard DELETEs exist only as annotated exceptions; secrets were leaked in its settings (anti-pattern to avoid). Scale the tooling down — principles, not the full 10-agent ceremony.
**Principle:** Inherit *verified* principles, not idealized ones; earn ceremony as the project grows.

## Decision 7: Adopt CodeRabbit + a scaled cr-local/fullpush push gate (2026-07-07)
**Context:** CodeRabbit is installed org-wide on `okpilot` (so it reviews every repo's PRs), and the `coderabbit` CLI is installed + Pro. The user requires cr-local before every push, per their LMS Plus `/fullpush` gate.
**Decided:** Adopt, scaled to this project: (a) `.coderabbit.yaml` (lean, Dart + SQL path_instructions, no secrets); (b) `.claude/commands/crlocal.md` — run `coderabbit review --base main --type committed`, triage apply/skip/defer reading source, min 2 rounds (3 for SQL/security), stop when ≥min and last round clean, ceiling 4 fix-commits; (c) `.claude/commands/fullpush.md` — analyze + test + build + crlocal + explicit push approval; (d) **branch per slice**, `main` stays green. CI/CD (GitHub Actions) added with Slice 1. _(Amended 2026-07-14 — the round counts here are **superseded by Decision 35**: cr-local now M=1/M=2; fleet floor 3/4, ceiling 6. cr-local's 4-fix-commit ceiling and "before every push" are unchanged.)_
**Principle:** cr-local before every push; the cloud bot on the PR is the authoritative gate. Earn heavier ceremony (multi-agent pipeline, e2e, scanners) as the project grows.

## Decision 8: Styling = stock Material 3 for now; theming deferred (2026-07-08)
**Context:** Coming from the TS/React world, the user styles with shadcn/ui. Flutter needs no such component library — Material 3 is built into the framework, and a "theme" is a `ThemeData` object (shadcn's CSS-variable block ≈ `ThemeData`; `ColorScheme.fromSeed(seedColor:)` turns one seed color into a full accessible light+dark palette).
**Decided:** Slice 1 (and until a slice actually calls for styling) uses **stock Material 3** — `ThemeData(useMaterial3: true, colorScheme: ColorScheme.fromSeed(seedColor: Colors.indigo))`. No component library, no custom theme. References bookmarked for the first styling slice (see IDEAS/NOTES): Material Theme Builder, `flex_color_scheme`, and `shadcn_ui`/`forui` if we ever want the flat shadcn aesthetic.
**Principle:** Good-looking defaults for free; earn a custom theme emergently, when a slice needs it.

## Decision 10: Run the backend locally for development; homebase deploy deferred (2026-07-08)
**Context:** Building the first real feature (Contacts) "with the backend". homebase was unreachable (SSH timed out) and the user said push nothing. Standing up the trimmed Supabase on homebase needs interactive Tailscale auth + a push — both off the table.
**Decided:** Run the trimmed stack **locally** in `backend/` via docker-compose — **Postgres + PostgREST + a Caddy gateway** that exposes the Supabase `/rest/v1` path so `supabase_flutter` works unmodified (mirrors the homebase shape). **GoTrue (auth) is deferred** to the first auth slice (database.md "Not now") — anon-role access under RLS for now. Dev secrets live in gitignored `backend/.env`; the Flutter client gets URL + anon key via `--dart-define-from-file`. Homebase deploy (into `okpilot/selfhost`, public HTTPS) is a deliberate later slice. **Soft-delete goes through a `soft_delete_contact` SECURITY DEFINER RPC** because a direct UPDATE of `deleted_at` fails the SELECT policy via PostgREST's RETURNING (42501) — a single-table exception to "direct under RLS", consistent with database.md's "soft-delete via function".
**Principle:** Develop against a faithful local copy of the real stack; promote to homebase when it's a deliberate, reachable step.

**Context:** With the Contacts feature working, stock Material 3 (indigo, tonal, rounded) looked "completely wrong" against the flat/tight/monochrome direction we'd converged on in the prototype (direction D + mono palette). This is the "theming slice" Decision 8 anticipated.
**Decided:** A single bespoke `ThemeData` in `lib/theme.dart` translating the prototype tokens — near-black ink as the *only* accent, `surfaceTint` transparent (kills M3 tonal elevation → truly flat), small radii (10/8px), hairline dividers, `VisualDensity.compact`, neutral-gray avatars, ink-filled buttons. **Both light and dark** first-class (`ThemeMode.system`). **Principle:** One theme file, one type scale — hierarchy from deliberate size+weight, never accident. Supersedes the "stock M3 for now" half of Decision 8.

## Decision 14: Backend deployed to homebase — tailnet-only; schema stays in the app repo (2026-07-08)
**Context:** Time to make the backend real. Chosen: deploy to homebase (Decision 10 revisited). Also settled the ongoing workflow so schema doesn't get duplicated across repos.
**Decided:** Deployed the trimmed stack to homebase as `okpilot/selfhost/stacks/firstapp-crm/` — **infra only** (Postgres 16 + PostgREST + Caddy gateway + PostgREST roles), fronted by the main Caddy at **`https://homebase.tail7ab4bc.ts.net:8452`** with a Tailscale TLS cert. **Tailnet-only** (the S23+ reaches it with the Tailscale app; no public internet exposure while there's no auth). Real secrets generated on the server into a gitignored `.env`; **started empty** (no seed). **The schema is NOT in selfhost** — `backend/migrations/` in *this* repo is the single source of truth, applied to any environment (local docker, homebase) by **`backend/deploy-homebase.sh`**, a forward-only migrator over the tailnet that tracks applied files in `public._migrations`. So a future change = new migration + UI here → test locally → run the migrator against homebase; selfhost is touched once and rarely again. App points at homebase via gitignored `dev-defines.homebase.json`.
**Principle:** Infra and schema are different things — the server is a dumb host (selfhost), the schema lives with the app and is applied to environments. No copy-paste between repos.
**Amendment (2026-07-09, PR #11):** The migrator's idempotency was broken — its per-migration exists-check ran `psql -c "…"` through `ssh → docker exec`, so the space-containing query got word-split on the remote side and always returned empty; the script re-applied *every* migration and only worked against a fresh DB (re-runs failed `relation … already exists`). Fixed to pipe the check over **stdin** with `psql -v :'name'` quoting (survives all three hops; the apply + ledger-insert paths already used stdin and were fine). The migrator is now genuinely idempotent — re-runs skip already-applied files. (Also: a homebase deploy may prompt a one-time Tailscale SSH re-auth; safe to let it continue or re-run after.)

The user has a mature lefthook + Claude-Code-hook setup in LMS Plus and wanted the equivalent here, in the same logic. ## Decision 18: Calendar events + attendees — date/time model + RPC write path (2026-07-09)
**Context:** Making the calendar real after the shell (Decision 16): create/edit/delete events and assign contacts as attendees. Prototyped in a throwaway artifact and approved; the build plan was run through **3 adversarial critics** (scope/YAGNI · correctness · design/UX) before coding, fixes folded in.
**Decided:**
- **Data model — `date` + `time` + `all_day`, NOT `timestamptz`.** No auth, single user, all local; the calendar's date logic is already local + DST-safe, so this dodges timezone complexity we don't need yet. **Trade-off (accepted + documented): single-day, non-overnight events only** — a `time`-only end with no end-date plus the `end_time > start_time` CHECK makes a 21:00→01:00 event impossible. Revisit with `timestamptz` if overnight/multi-timezone ever matter. Model stores times as `int?` minutes-from-midnight (pure Dart, no `TimeOfDay`), bridged to the picker at the UI boundary.
- **Attendees = a join table** `event_attendees(event_id, contact_id)` (composite PK, FKs `on delete cascade` — both entities are soft-deleted, so cascade is a safety net not the normal path). The `contacts` embed is **to-one** and comes back **null** for a soft-deleted attendee (RLS hides it); the client skips nulls.
- **Writes go through SECURITY DEFINER RPCs** (`create_event` / `update_event` / `soft_delete_event`), because an event + its attendees is a **multi-table** write (`database.md` #1/#2, atomic). They run as the table **owner → bypass RLS**, so `events`/`event_attendees` grant anon **SELECT only** (least privilege; no direct insert/update). **Corrected rationale:** the bypass is because SECURITY DEFINER runs as owner — NOT because the fn returns a scalar (the `soft_delete_contact` 42501 was a PostgREST REST-layer re-check, which doesn't apply inside a function). `update_event` **replaces** the attendee set; the hard-DELETE of join rows is the annotated exception to soft-delete-by-default (#4 — membership is derived, not a soft-deletable entity).
- **Reads = a direct embedded PostgREST select** (`events?select=…,event_attendees(contact_id, contacts(id,name,company))`), RLS-filtered, sorted in Dart (all-day first, then start time — `.order()` would put NULL/all-day last).
- **UI/theme:** the four views became data-driven; the timeline uses **greedy lane-packing** for overlapping events (improved on the plan's inset-only approach per the design critic — full occlusion hides data). New tokens in `theme.dart`: an `EventBlockStyle` ThemeExtension (fill **+ hairline border** + ink rail — border-defined so blocks read in light AND dark), plus mono `switchTheme` and `timePickerTheme`; the time picker is forced to **24-hour** (no AM/PM). Extracted a shared `InitialsAvatar` (with a `ring` param so stacked attendee faces separate in mono).
- **QA:** SQL curl-verified (RPCs, embed shape, all CHECK guards incl. overnight rejection); **31 unit + widget tests**; and a **visual QA on the Android emulator in light + dark** across every screen (the canonical run path — web has the known IPv6/origin backend issue).
**Principle:** Multi-table writes earn one atomic SECURITY DEFINER RPC; the model stays as simple as the single-user/local reality allows (date+time, not timestamptz); prototype → critics → build → visual QA.

## Decision 19: Event types — colour-as-data + a Settings home (design direction, 2026-07-09)
**Context:** Next feature: user-defined **event types** (categories) with colours, assignable to events. Explored in two throwaway artifacts (a quick concept, then a full-fidelity tap-through matching the real Day timeline / Agenda / form / detail), QA'd in-browser light + dark, and **approved by the user**. **Build not started** — data model + RPC + thin slicing go through adversarial critics before any Dart.
**Decided (design direction only):**
- **One type per event** — a category, not multi-label tags. `events` gains a **nullable** `type_id`; "No type" stays valid.
- **Colour is data, never chrome.** The app stays mono (theme.dart: "the accent IS the ink"); the *only* colour anywhere is a type's identity, surfaced as (a) a small filled **dot** in lists / pickers / detail / Agenda and (b) the **whole Day/3-day event block** coloured by type. Buttons/text/chrome stay ink. Linear/Attio discipline.
  - **Amended 2026-07-10:** the initial plan used a coloured **left rail** on the block (from the existing `EventBlockStyle.rail`); the user rejected it as an AI-slop tell ("accent bar on a card") — the design-principles skill lists the same cliché. Blocks are now **full-area coloured, no rail**. **Decided: Tint** (soft wash of the type colour + **ink** text) over Solid (bold fill + white text) — Solid was too loud for the mono system, risked white-on-amber contrast, and fought the quiet dot language; Tint keeps ink text primary and colour secondary. The `EventBlockStyle` extension will carry a **fill + text-colour** pair instead of a rail. **Corrected in Slice 3 (2026-07-10):** the extension keeps **fill + hairline border** (the no-type default); the *text* colour is derived in-widget, and a typed block swaps its fill for `tintForType(...)` while keeping the same border — so the border carries objecthood and colour is pure enrichment. `EventBlockStyle.rail` was removed outright.
  - The Event-types manager list shows **swatch + name only** — no per-type event count (user's call).
- **Month density dots are coloured by type** — the user's explicit call (overrode the initial "keep Month mono" proposal); no-type = neutral grey dot. **Busy days use "Deduped +N":** up to 3 dots for the *distinct types* present that day, then a `+N` count of the remaining events — chosen over capped-chronological (showed 3 identical dots for a day of meetings) and plain dedupe (hid busyness).
  - **Amended 2026-07-10 (Slice 3):** switched from deduped-by-type to **density dots, coloured** — up to 3 dots by *event count* (each coloured by that event's type; no-type → neutral ink), then `+N` = remaining events. Two adversarial critics caught that dedupe-by-type both **emptied no-type-only days** (0 distinct types → 0 dots) and **undercounted busyness** (3 same-type events read as one dot + "+2"). The user chose density-coloured. Out-of-month days stay grey (colour = "this month").
- **Curated palette, no freeform picker** — 9 muted mid-luminance swatches (blue · teal · green · amber · orange · red · purple · pink · slate) chosen to read on both `#FDFDFF` and `#121316`. Freeform RGB rejected as slop-prone.
  - **Amended 2026-07-10:** dropped to **8 swatches** — **slate removed** (a desaturated grey-blue collapses into the no-type neutral in both themes). Implemented in `lib/util/event_type_palette.dart` as named swatches (each carries a screen-reader label).
- **Management home = a new Settings destination.** Promote the nav to 3 tabs (Contacts · Calendar · Settings); **Settings → Event types** is the manager + editor, and types are *also* creatable on-the-fly from the event form's Type picker sheet. **Delete is non-destructive to events:** they keep their schedule and fall back to "No type" (can't be undone).
  - **Amended 2026-07-10 (Slice 3):** the form's Type picker is **pick-existing-only** — inline type-create was deferred (a scope-trim the user chose). The picker sheet offers the existing types, "No type", and a **"Manage types…"** action that opens the existing Settings manager; the list refreshes on return. Inline create-and-select is a tracked follow-up.
- **Implementation status (amended 2026-07-10):** the design was built in thin slices (supersedes this entry's opening "Build not started" note). **Slice 1** shipped — `event_types` table + `events.type_id` FK + `EventType` model + read embed (PR #13). **Slice 2** shipped — Settings → Event types manager/editor + `soft_delete_event_type` RPC, palette dropped to 8 swatches (PR #14). **Slice 3** shipped — `p_type_id` on the write RPCs, the event-form Type picker, full-area **tinted** Day/3-day blocks (no rail), dot + name in Agenda/detail/panel, coloured Month density dots + "+N"; a shared `TypeLabel` atom; `tintForType` (HSL-lightened + `alphaBlend` on dark) calibrated on the emulator light+dark (PR #15).

**Principle:** Introduce colour only as user-owned data in minimal tokens (a dot + a full-area block tint — the left rail was dropped, see the 2026-07-10 amendment above), never in chrome; give app config a real home (Settings) as the app matures; prototype → approve → critics → build.

## Decision 20: Split cloud-CodeRabbit handling into `/coderabbit` (triage) + `/replycoderabbit` (reply) (2026-07-10)
**Context:** Decision 17 adopted a single `/replycoderabbit` that both triaged and replied. In practice that conflated two moments — deciding (right after the review) and responding (a later sweep, sometimes a different session) — and a one-off "adaptation" this session quietly merged even more logic into it. LMS Plus keeps them as two commands. The redesign ran through **3 adversarial critic rounds (9 reports, verified against live PRs #14/#15)**, which killed several plausible-but-wrong approaches.
**Decided:**
- **Two commands, split by moment:** **`/coderabbit`** triages the cloud review (investigate each finding vs current source → FIX/DEFER/SKIP, using `/crlocal`'s rubric; implements FIX-NOW after approval; **never pushes**), then **`/fullpush`** pushes, then **`/replycoderabbit`** posts the reply (**never triages**). Wired into `/wrapup` in that order.
- **One shared `scripts/cr-findings.sh`** does fetch+normalize (the single source of truth for the hard-won CodeRabbit quirks), emitting the **union of ALL review runs**; each command applies its own logic. Fixture-tested; `[]`+exit 0 on clean, non-zero only on real failure.
- **Handoff = a machine-readable PR comment**, not context or a gitignored file. `/coderabbit` upserts a `<!-- crtriage -->` comment with hidden `<!-- crfinding:<id>:<verdict>:<ref> -->` lines; `/replycoderabbit` joins on `<id>` and upserts a `<!-- crreply -->` comment.
- **Stable finding identity** — a line-number-free content hash (lines move when code is fixed) that is the join/idempotency/loop-termination key; a re-raised already-disposed finding matches its prior verdict instead of bouncing the two commands forever. The **exact byte formula lives in `scripts/cr-findings.sh`** (its single source of truth): first 12 hex of `sha1(path + "\n" + normalized_title)`, where `normalized_title` is lowercased, trimmed, and internal-whitespace-collapsed (note the `\n` separator and the 12-char truncation — the ledger points at the script rather than restating the formula, to avoid drift).
- **Currency is NOT git-ancestry** (rebases make review commits diverge from head) and **NOT "latest review"** (actionable findings can sit in an older run than a later nitpick-only run). It's the union, validated against current source. Fix SHAs are resolved **live at reply time** (`git log`), never stored (rebases rewrite them).
**Refines:** Decision 17 (which stays as the "always answer the cloud bot" principle; this splits *how*).
**Principle:** One rulebook, two surfaces — triage before reply, decide once and record it durably on the PR, and anchor everything on a rebase-stable content identity, never on SHAs or line numbers.
- **Merged & dogfooded 2026-07-10 (PR #16 → `c2a3fc6`).** The first live `/coderabbit → /fullpush → /replycoderabbit` run was on the split's own PR, and it FIX-NOW'd two real bugs the pre-push `/crlocal` rounds had missed: **(1)** the comment lookups used an **unanchored** `test("<!-- crtriage -->")`, so a finding whose *description quotes* the literal marker string matched the wrong comment (I briefly overwrote the triage comment with the reply) → anchor all lookups to `^` (the marker is always the body's first line), and don't embed literal markers in human-facing finding text; **(2)** the `crfinding` payload was colon-delimited but the ref (a `fix: …` Conventional-Commit subject) contains colons → parse `id`+`verdict`+`remainder-of-line`, splitting on the first two colons only. Also fixed: invalid `gh api --jq --arg` (use `env.ME`), silent-drop of unmapped inline findings in `cr-findings.sh` (emit under a synthetic run + warn), crreply author-scoping. **Lesson:** eating our own dog food on the tooling PR is the cheapest place to find tooling bugs.

## Decision 23: Event comments — viewable soft-delete (2026-07-11)
**Context:** Adding comments on events so users can track metadata / decisions / follow-ups. Single-user, no author yet (~~auth lands later with issue #3~~ — auth is now WON'T-DO, **Decision 37**). Comments should stay readable even when archived — the UI shows a "Show archived" toggle to reveal them.
**Decided:**
- **Viewable soft-delete:** Unlike every other soft-deletable table, `event_comments`' SELECT policy is `using (true)`, NOT `using (deleted_at is null)`. Archived rows (`deleted_at IS NOT NULL`) stay READABLE so the app can surface them under a toggle.
- **No soft-delete RPC needed.** The normal pattern for soft-delete is a SECURITY DEFINER RPC (like `soft_delete_contact` / `soft_delete_event_type`) because a direct UPDATE of `deleted_at` fails PostgREST's RETURNING re-check against `using (deleted_at is null)` (42501 — row is gone after the mutation, so it can't pass the SELECT policy). Here, the SELECT policy is `using (true)`, so an archived row *survives* the re-check → **archive / unarchive / edit are all plain direct UPDATEs**, no RPC overhead. This is the *only* table with this SELECT-always policy; it is a deliberate, documented exception (database.md #4 amendment).
  - *Amended 2026-07-12 (Decision 26 / Slice 3):* the **write path** is now via the `create_comment` / `update_comment` / `soft_delete_comment` / `restore_comment` SECURITY DEFINER RPCs, for **uniformity** with every other entity — not because a 42501 forced it (this table's `using (true)` re-check argument above still stands; a direct write would still work). The `using (true)` **read** policy and the "archived stay readable" design below are unchanged. *(Precision on the "Why safe" line below: "never written by the client directly" means the **app** client — its archive/unarchive helpers now call the `soft_delete_comment` / `restore_comment` RPCs; a raw pre-auth **anon** PostgREST client can still `UPDATE deleted_at` directly, since the direct insert/update policies + grants are intentionally left in place until the auth-hardening sweep closes them under issue #3.)* **Amended 2026-07-15 (Decision 36 — pre-auth lockdown):** that direct anon write path is now **CLOSED** — `20260715120000_preauth_lockdown.sql` revoked the direct `insert, update` grants + dropped the `event_comments_insert/update` policies, so the RPCs are the sole write path here too (a raw anon client can no longer `UPDATE deleted_at` directly). The `using (true)` read policy is unchanged.
- **Why safe:** Because archived comments stay readable, the UI's visibility is unconditional (no "you've hidden this row" surprise). The `deleted_at` column is never written by the client directly (only via the repository's `archive` / `unarchive` helpers). The repository fetches by `event_id`, but that filter is **organizational, not an access boundary** — pre-auth, `select using (true)` lets anon read any comment via direct PostgREST, exactly the anon-permissive posture the other tables have. Owner/event-scoped access control ~~lands with auth (issue #3)~~ is **WON'T-DO** (Decision 37 — single-user + tailnet-only; anon-permissive is the accepted, permanent posture).
- **Implementation:** `event_comments` table (id, event_id FK, body, created_at, updated_at, deleted_at) under RLS. Single migration, direct-CRUD repository (CommentsRepository), self-contained UI section (_CommentsSection) on event detail. Comment model reads `deleted_at` back (unlike other models) so the UI can distinguish archived.
  - *Amended 2026-07-12 (Decision 26 / Slice 3):* the **repository is now RPC-based**, not direct-CRUD — all writes route through `create_comment` / `update_comment` / `soft_delete_comment` / `restore_comment` RPCs; reads stay direct (a plain select). The UI surface (`_CommentsSection`) and model (`Comment`) are unchanged.
**Principle:** Soft-delete is a convention with documented exceptions — this table's exception is justified by its read-always-archived design.

## Decision 25: Reload PostgREST's schema cache after every migration deploy (2026-07-12)
**Context:** Adding a comment failed on the S23+ ("Couldn't add comment"). Root-caused live: the `event_comments` migration *was* applied to homebase (table present, `is_insertable_into = YES`, `anon` has INSERT, RLS `with check (true)`), and the DB accepted the exact insert in a rolled-back `psql` test — yet `POST /rest/v1/event_comments` returned **404** (from `server: postgrest/12.2.3`) while `GET` returned `200`. Cause: **PostgREST reads the schema once at startup and caches it.** The homebase PostgREST had been running since *before* the migration; its DDL-watch had picked up the table for reads but not the writable set, so inserts 404'd. `deploy-homebase.sh` applied migrations but never told PostgREST to reload. Direct writes to older tables (contacts) worked; events dodged it by writing via RPC.
**Decided:**
- **`deploy-homebase.sh` sends `notify pgrst, 'reload schema';` after applying ≥1 migration** (piped over STDIN, not `-c`, to survive the ssh→docker→psql word-split — same reason as the exists-check). Zero-downtime. Fallback if PostgREST's LISTEN channel is ever disabled: `docker restart firstapp-postgrest`.
  - **Amended 2026-07-12 (RPC-migration Slice 0):** the **primary** reload mechanism is now migration `20260712120000_pgrst_ddl_watch.sql`, which installs the canonical Supabase `pgrst_ddl_watch` / `pgrst_drop_watch` event triggers. They fire `NOTIFY pgrst, 'reload schema'` on *every* DDL against a **running** PostgREST — including ad-hoc `psql` DDL applied outside `deploy-homebase.sh`, which the script's reload never covered. Verified live on homebase: a freshly-created function was callable via `/rpc/` with **no** manual reload, and `404`'d once dropped. **The script's `notify pgrst` was kept but changed** — the original `if [ "$applied" -gt 0 ]` guard on line above (conditional: only after ≥1 migration) was **replaced by a single *unconditional* line that runs on every deploy** — as a **cold-start safety net the triggers cannot cover**. **This supersedes the conditional `≥1 migration` rule stated on the line above — do not reintroduce the `if [ "$applied" -gt 0 ]` guard.** Rationale: on a from-scratch homebase, migrations 1..N build the schema *before* the trigger migration installs the triggers, and installing an event trigger does not itself fire `ddl_command_end` (empirically confirmed — nor does the ledger `INSERT`), so a fresh deploy emits zero `NOTIFY` of its own. Net roles: **triggers = running/steady-state + ad-hoc**; **script one-liner = fresh-DB cold start**. The `docker restart firstapp-postgrest` fallback still stands.
- **Immediate remediation** was the same `NOTIFY` run by hand against homebase; verified `POST` then returns a real `400` check-constraint error (route reaches the table) instead of `404`.
- **Diagnosis discipline that paid off:** verify the layer, don't trust the ledger — the migration ledger said "deployed", but the bug lived one layer up in PostgREST's cache. Reading the `server:` response header is what proved PostgREST (not Caddy) owned the 404.
**Related:** Exposed our error-handling blind spot — `catch (_)` discards the exception, so the phone logged nothing until a temporary `debugPrint` was added; proper fix tracked in **issue #23** (graceful for users, logged for us).
**Principle:** A migration isn't "deployed" until the layer that serves it knows about it. Schema-cache reload is part of the deploy, not an afterthought.

## Decision 26: RPC for all writes; reads stay direct (2026-07-12)
**Context:** We'd accumulated three write patterns — events via RPC (multi-table atomicity), contacts/event_types via direct writes + a soft-delete RPC, comments fully direct (Decision 23). After the schema-cache incident (Decision 25) prompted a review, we chose to standardize rather than keep deciding case-by-case. The decisive factor is **auth (issue #3)**: owner-scoping is far cleaner to add if every write already funnels through one function.
**Decided:**
- **All writes (INSERT / UPDATE / soft-delete) go through PostgREST RPCs** (`POST /rest/v1/rpc/*`). **Reads stay DIRECT** PostgREST `select` — an RPC read would throw away filtering / ordering / embedding / pagination (e.g. the calendar's events-with-types-and-attendees single call). "RPC-only" = **all writes**, not reads.
- **Why:** (1) **uniformity** — one write pattern to build, review, and reason about; (2) **auth pay-off** — when #3 lands, the `auth.uid()` owner check goes in *one* place per entity, not retrofitted across direct-write tables *and* functions; (3) **eliminates the RLS `RETURNING` re-check gotcha** (`42501`) that soft-delete on a `using (deleted_at is null)` table hits. *(Note 2026-07-15: motivation (2) is now moot — **Decision 37** made auth WON'T-DO, so no `auth.uid()` check is coming. Reasons (1) and (3) stand, and this decision remained worth it: the single RPC write path is exactly what let the **Decision 36** lockdown close the direct write door with zero app-behavior change.)*
- **Cost accepted:** more boilerplate (a function + migration + signature-versioning `drop/create` per write) and a modest refactor of the existing direct-write paths (contacts, comments) to RPCs.
- **Convention — one template per write RPC:** name `create_<entity>` / `update_<entity>` / `soft_delete_<entity>`; `security definer` with `set search_path = ''` hardening (#3); `revoke execute … from public`; `grant execute … to anon, authenticated` (owner checks arrive with auth). Change signatures via the documented `drop function if exists …; create or replace …` chain.
- **Relation to Decision 23:** revises the *write-path* posture going forward — comments/contacts writes move to RPCs. D23's `using (true)` SELECT-policy rationale for comments still stands (it governs *reads*, which remain direct).
**Principle:** Pick one write path and make everything follow it — the consistency compounds once auth and review enter.

## Decision 36: Pre-auth DB lockdown — the auth-independent subset of issue #3 (2026-07-15)
**Context:** Issue #3 ("DB security hardening") was filed to ship *with* the auth/GoTrue slice, but it bundles ~6 items across two buckets, and a meaningful subset needs **no** user identity. The client already writes exclusively via `SECURITY DEFINER` RPCs and reads direct-under-RLS (Decision 26 complete), yet every mutable table still granted anon a **second, unused** direct write door, and every RPC still carried Postgres's default `EXECUTE` to `PUBLIC`. Migration `20260715120000_preauth_lockdown.sql`. Verified on a fresh `postgres:15`: all 19 migrations apply clean; role-switched to `anon` — direct table writes denied, RPC writes succeed, the archived-task guard fires, `proacl` shows no PUBLIC entry on any of the 21 RPCs.
**Decided — three auth-independent hardenings (all *app*-behavior-preserving — the Flutter client only ever writes via RPC and reads direct, so nothing it does changes; the one deliberate behavior change is that a **raw anon direct-table write** now fails, which is exactly the boundary being closed; `events`/`event_attendees` already ran SELECT-only-for-anon + RPC-only-writes, proving the pattern):**
- **Close the direct anon write path** on the 5 still-open mutable tables (`contacts`, `event_types`, `event_comments`, `tasks`, `task_comments`): revoke the direct `insert, update` grants + drop the `<table>_insert`/`<table>_update` RLS policies. The RPCs (SECURITY DEFINER → run as owner → bypass RLS, no `FORCE ROW LEVEL SECURITY`) become the **sole** mutation interface. `grant select` + SELECT policies untouched (reads stay direct). This **subsumes** issue #3's "column-level write grants" item — anon can no longer write *any* table column directly, so it can't set `created_at`/`deleted_at`.
- **Revoke `EXECUTE … from public`** on all **21** client-facing SECURITY DEFINER RPCs (the explicit `anon, authenticated` grants persist). `pgrst_watch()` excluded (not SECURITY DEFINER, not client-facing).
- **`task_comments` archived-task guard:** the 4 task-comment RPCs now require the **parent task** be live (`create` pre-checks `exists`; `update`/`soft_delete`/`restore` AND an `exists` into the `where`). Closes the hole where a raw anon caller could mutate the log of a task the UI freezes (`CommentsSection.readOnly`). Deferred here from Slice 2b (Decision 33) precisely because closing one half while the write path stayed open wasn't a real boundary.
- **Remaining #3 scope** (see **Decision 37**, which followed): `auth.uid()` ownership checks + owner-based RLS are **WON'T-DO** (single-user + tailnet-only — not deferred); the only genuinely-open item is the **optional** `SET search_path = ''` + schema-qualify convention hardening (a separate ripple-heavy slice, kept out to keep this one *app*-behavior-preserving).
**Rule-reversal-sync:** this reverses the "direct write path open / PUBLIC-execute deferred to #3" posture asserted across the ledger (Decisions 15/23/33 amended in place, dated), `database.md` (#2/#4), the 5 sibling migration headers (dated "superseded by 20260715120000" notes — comment-only, executable SQL untouched), `.coderabbit.yaml`, and the `db-security-reviewer` / `red-team` fleet guidance. `backend/README.md` carries the linchpin verify curls.
**Principle:** Ship the security win you *can* close now — least-privilege that needs no identity shouldn't wait on the auth slice; leave only what genuinely needs a user behind the deferral.

## Decision 37: No login — single-user + tailnet-only is the security boundary; the auth half of issue #3 is WON'T-DO (2026-07-15)
**Context:** Issue #3's remaining items after the Decision 36 lockdown were `auth.uid()` ownership checks + owner-based RLS — both of which assume **multiple users** and a **reachable-by-strangers** API. Neither holds here: this is the owner's personal CRM, a **single user**, and the API is **tailnet-only** (Tailscale TLS, `homebase.tail…ts.net`; not exposed to the public internet — Decision 14). The owner explicitly does **not** want a login.
**Decided:**
- **No GoTrue / no login.** The security perimeter is **Tailscale** (only devices on the owner's private tailnet can reach the API) plus **single-user** (there is no user-B to scope away from user-A). Under that model, `auth.uid()` owner checks and owner-based RLS would enforce a distinction that does not exist — pure friction, no gain. The anon-permissive posture (anon can read/write live rows *via the RPCs*, over the tailnet) is the **accepted** posture, not a temporary gap.
- **Issue #3's auth items are WON'T-DO** (not "deferred"): the `auth.uid()` ownership check + owner-based RLS lines are closed as out-of-scope for this project's threat model. What remains of #3 is only the **optional** `SET search_path = ''` defense-in-depth hardening (a small slice, do-if-desired — it needs no auth).
- **The Decision 36 lockdown still stands and was still worth doing** — it is **not** auth. Revoking PUBLIC execute, closing the direct write path (RPC = sole write path), and the archived-task guard protect **data integrity** (the app's validation/guards can't be bypassed by a stray script or future integration) regardless of who is calling. Good hygiene even for a solo, tailnet-only CRM.
- **Accepted residual risk (stated, not hidden):** with no login, any device *already on the tailnet* — or the API if it were ever misconfigured to be publicly exposed — talks to the DB as the anon role with full read/write. A login would be a safety net for a compromised-tailnet-device or an accidental-exposure scenario. The owner accepts this given trusted devices + a private tailnet. **If the CRM is ever shared with another person, exposed beyond the tailnet, or made multi-tenant, revisit — auth becomes necessary then**, and this decision is the place that flip is recorded.
**Principle:** Match the security work to the actual threat model. A single-user, tailnet-only app's perimeter is the tailnet; bolting on per-user auth enforces a distinction that doesn't exist. Say "won't-do" out loud rather than carrying a permanent "deferred" that never comes.

## Decision 38: Task importance — a fixed 0..3 priority marker (! / !! / !!!) (2026-07-15)
**Context:** Tasks had no notion of priority — a long active list gives every task equal weight. The owner asked for an importance tag, "like !, !!, !!!". The design was prototyped light+dark before any Flutter (feedback-prototype-first) and two forks were the owner's call: **coloured** marks (not mono) and **sort by importance** (not just a passive tag).
**Decided:**
- **Data:** an `importance smallint not null default 0 check (importance between 0 and 3)` column on `tasks` (0 = none, 1/2/3 = !/!!/!!!). A **fixed semantic scale**, deliberately **not** Decision 19 colour-as-data — Decision 19 is *user-owned* colour (event-type swatches the user picks); importance-level→hue is fixed chrome the user can't recolour. It borrows the amber/orange/red **hues** from `kEventTypePalette` for one coherent colour language, but tuned per-theme for legibility as text (raw `#C6952F` amber fails contrast on the light surface → light darkens each hue, dark lightens it — `lib/util/importance.dart`).
- **Display:** **bare bold coloured marks** (owner's pick over a tinted-pill / solid-badge — the calmest of the three), one shared `ImportanceMarks` widget, on the list rows (muted for done/archived so active urgent tasks stay loudest) and beside the detail's status pill. The `!` glyph carries the signal; colour is enrichment, and a `Semantics` label ("Importance High") spells the level out — the marker is never colour-alone (design-principles a11y, as `TypeLabel`).
- **Sort:** active tasks sort **highest-importance first**, then newest (`fetchAll` gains `.order('importance', ascending:false)` between the `is_done` and `created_at` orders). Completed/archived groups sort the same way — a harmless side effect of one global order.
- **Form:** a custom segmented **None / ! / !! / !!!** picker (not M3 `SegmentedButton`, whose tonal fill fights the mono theme), between Notes and People; default None.
- **RPCs:** `create_task` / `update_task` gain `p_importance`, via the drop+recreate signature-change convention. `update_task.p_importance` is **REQUIRED (no default)** — same defensive rule as `p_contacts`: an omitted arg would silently reset the marker, so a stale caller fails loud (PGRST202) instead. `create_task` keeps a default (a create has nothing to reset).
- **Lockdown upheld (Decision 36):** dropping the old functions discarded their PUBLIC-execute revokes, and a recreate hands back default PUBLIC execute — so the migration **re-issues both** the `revoke execute … from public` and the `grant … to anon, authenticated` on the new signatures. Verified on a fresh DB: `public` execute = false, `anon`/`authenticated` = true. This is now the general rule for any post-lockdown drop+recreate of a client-facing RPC (`database.md` #2).
**Principle:** A fixed semantic scale is chrome, not data — reuse the app's existing hue language for coherence, but tune it for its new job (legible text, not a swatch) and keep the signal in the glyph so colour never rides alone. When a slice recreates a locked-down RPC, re-assert the lock in the same migration — a recreate silently reopens PUBLIC execute.

## Decision 39: Task categories — a separate taxonomy, user-owned colour swatches (Slice A, 2026-07-15)
**Context:** Users need to categorize their tasks, just as they categorize events with event types. The product choice: tasks get their **own** independent colour list (not shared with event types), so a task category never clutters the event Type picker and vice versa.
**Decided:**
- **`task_categories` table** — mirroring `event_types`: `id`, `name` (non-blank check), `color` (6-digit #RRGGBB, DB-authoritative, portable), `created_at/updated_at` (via trigger), `deleted_at` (soft-delete). RLS with `using (deleted_at is null)` for SELECT. **POST-LOCKDOWN: no direct insert/update RLS policies or grants** — writes go exclusively through the 3 SECURITY DEFINER RPCs (Decision 36), cleaner than event_types which shipped direct writes and closed them later. The RPCs: `create_task_category(p_name, p_color)` → `uuid`, `update_task_category(p_id, p_name, p_color)` → `uuid` (with deleted_at guard + `no_data_found` raise), `soft_delete_task_category(p_id)` → `void`. Each gets the Decision 36 lockdown discipline: PUBLIC execute revoked, anon/authenticated granted.
- **Separate list, same Palette:** Task categories use the **same 8-swatch colour palette** as event types (`lib/util/event_type_palette.dart` — amber/teal/green/orange/red/purple/pink/blue, chosen for contrast on light + dark surfaces, no-type → neutral grey). Slice A **reuses** the existing `TypeSwatch` colour-dot atom because the palette and colour-picker UI are identical; only the parent entity differs.
- **Slice A (committed) — the entity + Settings manager only.** New `TaskCategory` model (id/name/color); `SupabaseTaskCategoriesRepository` (fetch all, create, update, soft_delete); a new `TaskCategoriesScreen` under Settings (create/rename/recolour/delete inline, palette swatch-grid to choose a colour, reuses the shared palette + `TypeSwatch`). No task references a category yet.
- **Slice B (next) — the task↔category join:** the product choice is **many-per-task**, so Slice B adds a `task_category_links` join table (mirroring `task_contacts`, many-to-many), threads a `p_categories uuid[]` param onto the task write RPCs (drop+recreate), a `Task.categories` model field, a multi-select chip picker on the task form (mirroring the People picker), and colour labels on task rows + detail (this is where `TypeLabel` gets reused).
**Principle:** Repeat a proven shape (event types) for a new axis; separate lists keep the taxonomy breathing room. Slice the entity first, then the join — let Slice A ship before reaching for the relational complexity.

## Decision 40: Task categories — the task↔category join (Slice B, 2026-07-15)
**Context:** Decision 39 (Slice A) stood up the `task_categories` entity + its Settings manager, but no task referenced a category. Slice B closes the loop: a task carries **many categories** via a join table, chosen on the task form, shown on list rows + detail.
**Decided:**
- **`task_category_links` join table** — verbatim mirror of `task_contacts` (20260714160000): composite PK on `(task_id, category_id)`, both FKs `on delete cascade`, reverse index on `category_id` (so "which tasks have this category?" doesn't scan). No `deleted_at` or timestamps — membership is derived, not a soft-deletable entity with its own history (the annotated exception, database.md #4). RLS with `using (true)` SELECT policy (archived tasks stay embeddable; a soft-deleted category → embedded null when RLS hides it). No write policy/grant — membership set only by the task-write RPCs.
- **RPC signature change — `p_categories uuid[]` parameter:** `create_task(text, text, uuid[], smallint, uuid[])` and `update_task(uuid, text, boolean, text, uuid[], smallint, uuid[])` recreated via the drop+recreate pattern (old signatures dropped to avoid PGRST203 overload; the Decision 36 lockdown-revoke + anon/authenticated grant re-issued on the new signatures; identical behaviour otherwise). `p_categories` is a `uuid[]` with default `{}` for create (a create has nothing to wipe); **`p_categories` is REQUIRED (no default) on update** — same defensive rule as `p_contacts` and `p_importance` (an omitted arg would silently clear the category set → a stale caller gets PGRST202 function-not-found instead). Insertion via `unnest(p_categories) … on conflict do nothing` dedupes. Deletes on update are hard-DELETE of join rows (derived-membership exception).
- **Model + Repository:** `Task` gains a `categories` field (List<TaskCategory>, id/name/color, null when joined category was soft-deleted + hidden by RLS). `Task.fromJson` skips the nulls. `Task.copyWith(categories: …)` preserves it. `Task.toRpcParams()` sends `p_categories: categoryIds`. `TasksRepository` threads the list through both read and write paths. All six task-reconstructing-fake paths thread categories (`create` + `update` + `archive` + `restore` + two fetch variants — `Task(…)` + `fromJson` already do it; the fakes need the field injected). Every `toRpcParams() map` assertion checks categories' presence in the spread.
- **UI:** the task form gets a **category picker section** (generalized from the contact picker; the sheet lists all live categories + a "No categories" empty state + an inline "Manage categories…" → Settings shortcut); multi-select chips (reuse `TypeSwatch` atom, one chip per selected category). The task detail shows **coloured category labels** (reuse `TypeLabel` atom — dot + name, no tap yet). Narrow phone and wide desktop both apply (detail pane mirrors full detail). Row visual: **inline coloured dots** (the same `TypeSwatch` renders on-demand for each category, shrunk to a tiny dot) after the importance marker, empty when none.
- **Scope / stated skips:** two-phase creation (create-category + add-to-task in separate steps) is not offered in this slice; user creates categories first via Settings, then links them on the task form. No remove-one-category-from-picker UX (user re-saves the form).
**Principle:** Repeat a proven join pattern (task contacts / event attendees) for a new entity; mirror the shape so data and UX are instantly recognizable. The field is threaded through all reconstructing fakes, not just the update path — the rule learned (count 4: notes → contacts → importance → categories).
**Deploy note:** migrations + app ship together (PR + `/fullpush` + `/crlocal` + cloud CR + deploy); installed clients rebuild to send `p_categories` (both `create` and `update` calls change arity). *(Amended 2026-07-16 — **SHIPPED & DEPLOYED:** PR #47 → squash `37a3e37`; migration `20260715150000` live on homebase (SSH-verified: new 5-/7-arg RPC sigs, PUBLIC execute revoked, `task_category_links` RLS anon SELECT-only); `/updatelinux` done; `/updatephone` owed. Cloud CR on #47: 6 findings → 5 FIX `6e1a230` + 1 SKIP, bot confirmed all in-thread.)*

## Decision 41: Idempotent create_* RPCs — client-supplied id + ON CONFLICT (issue #9, 2026-07-16)
**Context:** CodeRabbit flagged on PR #8 (and re-raised on `create_event_type` in Slice 2) that every `create_*` minted the id server-side (`gen_random_uuid()`) then the repo did a **separate** `_fetchOne`. If the response drops after the row commits — or a user re-taps a hung "Add" — a retry inserts a **duplicate**. Deferred as #9; by 2026-07-16 the window was open on **all 7** create RPCs (`create_event`, `create_contact`, `create_event_type`, `create_task`, `create_task_category`, `create_comment`, `create_task_comment`) and the retrofit cost grew with each new entity, so it was paid in one coordinated slice rather than the earlier "one-per-slice" plan.
**Decided:**
- **RPC template (one migration `20260716120000`, drop+recreate all 7):** each `create_*` gains a trailing optional `p_id uuid default null`, computes `v_id := coalesce(p_id, gen_random_uuid())`, inserts the id **explicitly** with `on conflict (id) do nothing`, and returns `v_id`. The repo's existing `_fetchOne(v_id)` returns the row whether just-inserted or pre-existing → a same-id retry is a no-op. `p_id` is appended **last** (PostgREST binds by name, so order is irrelevant; a trailing default keeps every signature valid). Junction inserts (event_attendees, task_contacts, task_category_links) retain `on conflict do nothing` and now run only when the parent insert succeeds (`if found`), keeping same-id replays pure no-ops (they can't union a changed membership set onto an existing/archived row). **Decision 36 lockdown re-issued** on every new signature (drop discards the ACL): `revoke execute … from public` + `grant … to anon, authenticated`. Ends with `notify pgrst, 'reload schema'` (signature change; deploy-homebase never reloads PostgREST).
- **`create_task_comment` special case:** its archived-parent guard is folded into the insert (`insert … select … where exists (parent live) … on conflict (id) do nothing`) — race-safe vs a concurrent `soft_delete_task`. Since the insert now affects zero rows for **two** reasons (guard rejected OR idempotent replay), the old `if not found` is replaced by a post-insert `if not exists (select 1 from task_comments where id = v_id) then raise` — so a replay returns the existing comment and only a genuine archived/missing parent raises (`no_data_found`). Correct even in the "created live, parent archived between attempts, retried" case.
- **Client:** new `lib/util/ids.dart` `newEntityId()` (adds the `uuid` package) is the single mint point. Each model's `.draft` becomes an id-minting `factory` (`id ?? newEntityId()`); `id` now rides in `toRpcParams()` (so the 4 spread-`update` repos drop their now-redundant explicit `p_id` — the create-shaped map already carries it and matches `update_*`'s param set). `Comment` (no `toRpcParams`) gets `p_id` at the two repo `add()` call sites. Each of the **6 form surfaces** holds a stable `_pendingId` reused across save re-taps: the 5 pop-on-success forms use `late final` (route discarded on pop → fresh id next open); the `CommentsSection` composer stays mounted, so its `_pendingId` is **mutable** and reset after each successful add (else the next comment conflict-skips against the row just written). Contact/event forms build via the main constructor (`id: widget.existing?.id ?? _pendingId`), not `.draft`.
- **Scope / stated skips:** `update_*`/`soft_delete_*` unchanged (retrying re-applies the same values — already idempotent). Event comments keep no live-parent guard (view-archived is a feature). The Comment **test-fakes** keep synthetic monotonic ids (`c${_seq++}`) — they proxy `created_at` ordering via `sort((a,b)=>b.id.compareTo(a.id))`, so honoring `draft.id` there would scramble order; entity fakes were updated to persist `draft.id`.
**Principle:** Fix the template, not just the instances — codified in `database.md` rule #2 so `create_*` #8 inherits idempotency and `coderabbit-sync` enforces it. When the retrofit cost grows per-slice (unlike a pure deferral), pay it once as a coordinated slice.
**Deploy note:** migration + app ship together (all 7 create calls change arity; the client now sends `p_id`). Installed clients must rebuild — an old client calling the pre-`p_id` signature would 404 (PGRST202) after the drop+recreate. *(Amended 2026-07-16 — **SHIPPED & DEPLOYED:** PR #48 → squash `7016337`; migration `20260716120000` live on homebase (SSH-verified: all 7 `create_*` at `+p_id` arity, PUBLIC execute revoked / anon granted). cr-local (3 rounds) added the `if found` junction-gating; cloud CR on #48: 4 findings → 3 FIX `b35de77` + 1 DEFER (#49, the task_comment `FOR UPDATE` race), all replied in-thread. `/updatelinux` done; **`/updatephone` owed**.)*

## Decision 42: Shared test fakes in `test/support/fakes.dart` (issue #10 item 1, 2026-07-16)
**Context:** CodeRabbit flagged on PR #8 (deferred as #10) that the hand-written private fake repositories were copy-pasted across test files. By 2026-07-16 the duplication had grown from 3 files to **13** — every `abstract interface class` repo (contacts, events, event_types, task_categories, tasks, comments) had its happy-path fake reproduced in file after file, so adding a method to any repo interface meant editing N parallel copies (the same blast radius the CLAUDE.md "every reconstructing fake" rule keeps catching). Test-fake completeness is opaque to `flutter analyze`, the hooks, and CodeRabbit, so the drift only ever surfaced as a widget-test failure.
**Decided:**
- **One shared file, `test/support/fakes.dart`,** holding 9 PUBLIC fakes (dropped the leading `_` so they import): `FakeContactsRepo` / `FakeEventsRepo` (+ `lastCreated` capture) / `FakeEventTypesRepo` / `FakeTaskCategoriesRepo` (each seedable via a positional `[seed = const []]`), `FakeTasksRepo` (inert echo), `StatefulTasksRepo` (persists mutations; superset captures `lastUpdated` + `archivedId` + `restoredId` so it serves both the list and detail tests), `ThrowingTasksRepo`, `FakeCommentsRepo` (inert) and `SeededCommentsRepo` (filter-by-parentId + sort + `c$seq` add). Fakes `implements` (never `extends`) the interfaces — matches the existing convention; no `SupabaseClient` needed. Test files import `'support/fakes.dart'`.
- **Governing boundary — a fake is shared only once its body is duplicated across ≥2 files.** Single-file specials stay local: `_FailingContactsRepo`, `_FailingEventsRepo`, `_FakeTypesRepo`/`_FakeCategoriesRepo` (full-CRUD stateful), `_RefreshFailsRepo`, `_OrderedRepo`, the two divergent `_FailingCategoriesRepo`, `_CreateFailsRepo`, `_RecordingTasksRepo`, `_FailingRepo`, `_UpdateFailsRepo`, `_FlakyRecordingTasksRepo`, and comments_section's stateful `_FakeCommentsRepo` (throwOnFetch / archiveGate / real edit). Moving these would either pollute the shared happy-path fake or add indirection for zero dedup win. Net −475 lines across the test files (379 insertions, 854 deletions, commit `a08c199`); 229 tests stay green.
- **Two comments fakes, not one:** the inert `add` echoes the draft while the seeded `add` rebuilds `Comment(id:'c$seq')` — a real behavioral difference, so they stay two honest fakes rather than one leaky superset. The same private name (`_FakeCommentsRepo`) had meant the SEEDED tier in `task_detail` but the INERT tier elsewhere; the migration mapped each call site by behavior (task_detail → `SeededCommentsRepo`), not by name (plan-critic caught the drop-underscore trap).
**Principle:** Extract a test helper once it's duplicated ≥2×, not on first sight; keep single-file behavioral specials local. Shared happy-path fakes shrink the blast radius of a repo-interface change and give the "thread every field through every reconstructing fake" rule one file to check instead of N.
