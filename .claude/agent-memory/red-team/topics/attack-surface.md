# Attack-surface matrix — red-team (First Android App)

> **Protected topic file — never pruned, never inlined into MEMORY.md.** The threat-vector →
> coverage matrix. Update it in place after every red-team review: add new vectors the diff
> introduces, update `Covered by` when a check is recommended/lands, and transition `Status`.
> `Status` legend: `INFO pre-auth` (expected #3) · `gap` (reachable, no check) · `partial (static
> only)` (a static check exists but no runtime curl/widget/integration one) · `covered` (a
> curl/widget/integration check exists) · `pending (auth #3)` (goes live when auth lands).

## Current vectors (pre-auth — seeded 2026-07)

| Vector | Surface | Covered by | Status |
|---|---|---|---|
| anon full CRUD over live rows | `anon` can read/insert/update/delete any non-deleted row via PostgREST, bypassing the RPCs | anon-scope curl (recommended, documents posture) | INFO pre-auth (#3) — expected, not an attack |
| RPC EXECUTE granted to PUBLIC | Postgres default `EXECUTE` to `PUBLIC` on `SECURITY DEFINER` RPCs; `grant … to anon` is additive, not a lock-down | `db-security-reviewer` ISSUE at the gate; `revoke execute … from public` sweep | INFO pre-auth (#3) — its sweep, not a red-team attack |
| Soft-delete → hard-delete | Three `soft_delete_*` RPCs live (contact, event, event_type) — all `update … set deleted_at = now() where … deleted_at is null`, no `DELETE`. Must flag, not erase | soft-delete-doesn't-hard-delete curl: after RPC, `select … where id=…` still returns the row with `deleted_at` set — **NOT yet written** (backend/README.md has only a contacts `select` curl) | **gap** — recommend the survives-with-deleted_at curl for all three RPCs |
| Soft-deleted type → dangling embed | Soft-deleting an `event_type` must make `events?select=…,event_types(…)` embed `null` (type_id FK has NO on-delete; row hidden by SELECT policy, not gone) | Client tolerance **covered** by `test/event_test.dart` (null embed → `Event.type == null`, lines 104-122; null contact embed skipped, lines 54-63). DB-layer producing the null: **NOT** documented by curl | **partial** — client covered; recommend the DB embed-returns-null curl |
| RLS present on new table | A new `create table` reachable by `anon` without RLS = unfiltered exposure | `db-security-reviewer` static check; red-team confirms the *surface* | **partial (static only)** — RLS presence is verified statically; no anon-RLS *runtime* curl exists yet (`covered` in this matrix means a curl/widget/integration check) |
| Cross-user data access | User A reads/mutates user B's rows | — | pending (auth #3) — flips to CRITICAL/ISSUE when auth lands |
| Owner-scoping | RPC/policy scopes rows to the caller (`auth.uid()`) | — | pending (auth #3) — flips to ISSUE when auth lands |
| event_attendees hard-DELETE via update_event | `update_event` runs `delete from event_attendees where event_id = p_id` to replace the set (annotated exception, database.md #4 — membership is derived). Guarded: bails with `no_data_found` if the event is unknown/soft-deleted, so a hidden event's set can't be silently rewritten | The `if not found` guard is the control; no curl asserts it. Pre-auth anon can already call it | INFO pre-auth (#3) — intended design + guarded; not an attack today |
| event_comments SELECT `using(true)` (archived readable) | Deliberate divergence: SELECT policy is `using(true)`, so archived (`deleted_at IS NOT NULL`) rows stay readable — that IS the "Show archived" feature. Columns are id/event_id/body/timestamps only — NO author/identity/auth column, so no extra sensitive exposure beyond comment bodies vs the anon-CRUD baseline | Widget round-trip `test/comments_section_test.dart`; live curl run in commit 3a87cc8 (archived row still SELECTable) but NOT recorded in backend/README.md | INFO pre-auth (#3). **Phase-flip watch:** post-auth, `using(true)` + no author column means once an author col is added a user could read another user's archived comments — revisit owner-scoping for this table when auth lands |
| event_comments archive/unarchive/edit via direct UPDATE `using(true) with check(true)` | One UPDATE policy covers edit/archive/unarchive; anon can mutate any comment (expected pre-auth). Non-destructive: NO delete grant, and body CHECK `length(trim(body))>0` blocks blanking. Column grants aren't restricted, so anon could also rewrite `event_id`/`created_at` — integrity nit, subsumed by anon-CRUD baseline | This is the ONE table where a plain anon `select … where id=` PROVES soft-delete persistence (archived rows are visible here, unlike the RPC tables filtered by `deleted_at is null`). Curl run in commit but not recorded | INFO pre-auth (#3) — recommend recording the archived-persists-with-deleted_at curl in backend/README.md so "covered" is durable |
| event_comments `on delete cascade` from events | FK `event_id … on delete cascade` is a hard-delete path for comments. Dormant today: events are NEVER hard-deleted (verified — no `delete from events` anywhere; `soft_delete_event` only sets `deleted_at`), so the cascade never fires | Guarded implicitly by the events soft-delete-only invariant (database.md #4). No direct check | INFO pre-auth (#3) / **latent-coupling watch** — if a hard DELETE of events is ever introduced, comments would be silently hard-deleted, violating soft-delete-only. Keep "events are never hard-deleted" as the control |
| contact write RPCs `create_contact` / `update_contact` (SECURITY DEFINER) | New write path added `20260712130000` (Decision 26). `search_path=public` pinned; bodies are parameterized static INSERT/UPDATE with `trim`/`nullif` — NO dynamic SQL, no injection. `update_contact` guards `where id=p_id and deleted_at is null` + `raise no_data_found` if `not found`: cannot mutate/resurrect a soft-deleted or absent row (never writes `deleted_at`, WHERE excludes hidden rows). No net new surface pre-auth — create/update touch only non-delete fields (deleted_at stays null), so a direct write under the still-present `contacts_insert`/`contacts_update` policies would do the same. Name CHECK `length(trim(name))>0` fires through the RPC (blank/NULL name → check/not-null violation, DB catches) | `update_contact`-refuses-soft-deleted curl (soft_delete_contact a row → update_contact it → expect `no_data_found`), plus create_contact happy-path (trim + empty→null) and blank-name-rejected curls, **RECORDED in `backend/README.md`** ("Verify: contact write RPCs", commit 13a0b3e); also exercised as the `anon` role on a fresh postgres:16 during the slice | **covered** — the three curls are durable in `backend/README.md`. Severity was INFO (correctness, not an attack pre-auth). **Phase-flip watch (auth #3):** these RPCs bypass RLS, so once #3 tightens contacts policies to owner-scoping, the `auth.uid()` owner check MUST be added *inside* create/update_contact or the RPC path lets a user write any row — this is the single spot the header comment flags |
| event_type write RPCs `create_event_type` / `update_event_type` (SECURITY DEFINER) | Near-clone of the contact write RPCs, added `20260712140000` (Decision 26, Slice 2). `search_path=public` pinned; bodies are parameterized static INSERT/UPDATE with `trim(p_name)` (color passed through as-is) — NO dynamic SQL, no injection. `update_event_type` guards `where id=p_id and deleted_at is null` + `raise no_data_found` if `not found`: cannot mutate/resurrect a soft-deleted or absent row (never writes `deleted_at`, WHERE excludes hidden rows). No net new surface pre-auth — create/update touch only name/color (deleted_at stays null), so a direct write under the still-present `event_types_insert`/`event_types_update` policies would do the same. Both table CHECKs (`length(trim(name))>0`, `color ~ '^#[0-9A-Fa-f]{6}$'`) re-fire through the RPC | `update_event_type`-refuses-soft-deleted curl (soft_delete → update → expect `no_data_found`), create happy-path (trim proven `"  Focus  "`→`Focus`), blank-name→400 and malformed-color→400 curls, **RECORDED in `backend/README.md`** ("Verify: event type write RPCs") — both table CHECKs (name + color) proven firing through the RPC | **covered** — the curls are durable in `backend/README.md`. Severity INFO (correctness, not an attack pre-auth). **Phase-flip watch (auth #3):** same as contacts — these RPCs bypass RLS, so when #3 owner-scopes event_types the `auth.uid()` owner check MUST go *inside* create/update_event_type or the RPC path lets a user write any type |
| DDL-watch event triggers (`pgrst_ddl_watch`/`pgrst_drop_watch` → `public.pgrst_watch()`) | Two event triggers fire `NOTIFY pgrst, 'reload schema'` on any DDL (`20260712120000`). NOT attacker-reachable: event triggers fire ONLY on DDL, which anon/authenticated cannot run (no CREATE/ALTER/DROP via PostgREST — DML/RPC don't fire them). Function returns `event_trigger` → not exposable as an RPC and errors if invoked outside trigger context, so the usual PUBLIC-EXECUTE #3 concern is doubly moot. Channel/payload are fixed literals (no user data interpolated) → no NOTIFY spoof/injection; only listener is PostgREST, reload is idempotent. Not SECURITY DEFINER (runs in DDL executor's context — no elevation) + `search_path=''` (no CVE-2018-1058 surface). Postgres dedupes identical NOTIFY within a txn → a multi-statement migration = one reload, no self-storm | No attacker vector to cover. **Availability coupling** = the only real consideration: any future edit that makes `pgrst_watch()` fallible would abort EVERY DDL (incl. migrations). Recommend a one-time smoke check on homebase (as postgres): `psql -c "create table _z(); drop table _z;"` succeeds AND PostgREST picks up a new object without a manual reload | INFO pre-auth — no anon-reachable surface. **Latent-coupling watch:** keep `pgrst_watch()` body a bare NOTIFY; if it ever grows logic, re-review the DDL-abort blast radius |
